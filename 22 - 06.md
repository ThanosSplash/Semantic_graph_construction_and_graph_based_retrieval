# Report προόδου σε σχέση με το προηγούμενο feedback

Η νέα έκδοση δείχνει καθαρή πρόοδο. Έχουν γίνει αρκετές από τις διορθώσεις που ζητήθηκαν, έχουν προστεθεί νέα κομμάτια πειραματικής υποδομής και η υλοποίηση είναι πιο κοντά σε κανονικό experimental project.

Ωστόσο, υπάρχουν ακόμη μερικά σημαντικά bugs και ασυνέπειες που πρέπει να διορθωθούν πριν αρχίσουν τα τελικά experiments.



# 1. Τι έχει βελτιωθεί

## 1.1 Έγιναν toy tests για τα metrics

Έχουν προστεθεί tests για απλά παραδείγματα με `predictions` και `ground_truth`, κάτι που είναι σωστό βήμα για να ελεγχθούν Recall, RR, nDCG και MAP.

Αυτό είναι θετικό, γιατί πλέον δεν βασιζόμαστε μόνο στο ότι “ο κώδικας φαίνεται σωστός”, αλλά υπάρχει προσπάθεια επαλήθευσης με μικρά παραδείγματα.

Προσοχή όμως: σε κάποια χειροκίνητα παραδείγματα θέλει ξανά έλεγχο η αριθμητική. Για παράδειγμα, αν:

```text
Prediction = ["A", "B", "C", "D"]
Ground truth = ["C", "X"]
```

τότε το πρώτο relevant είναι το `C`, που βρίσκεται στη θέση 3. Άρα:

```text
RR = 1/3
```

όχι `1/2`.



## 1.2 Το RR βελτιώθηκε

Η `RR_score` πλέον επιστρέφει και score και rank/index:

```python
return 1/(indx + 1), indx
```

Αυτό είναι καλό, γιατί επιτρέπει query-level analysis, δηλαδή να βλέπουμε σε ποια θέση εμφανίστηκε το πρώτο relevant passage.



## 1.3 Το nDCG βελτιώθηκε σε σχέση με πριν

Η προηγούμενη υλοποίηση έδινε λάθος arguments στη `ndcg_score`. Τώρα χρησιμοποιείται:

```python
relevance = np.array([[1 if p in correct_set else 0 for p in predictions]])
mock_scores = np.array([[len(predictions) - i for i in range(len(predictions))]])

return ndcg_score(relevance, mock_scores, k=k)
```

Αυτό είναι καλύτερο από πριν, γιατί το `y_true` είναι πλέον το relevance vector και το `y_score` είναι ένα descending score που διατηρεί τη σειρά του ranking.

Ωστόσο, για retrieval setup με πολλαπλά relevant documents, πιο καθαρό και ασφαλές παραμένει ένα χειροκίνητο `ndcg_at_k`, ώστε το ideal DCG να λαμβάνει υπόψη όλα τα relevant documents μέχρι το K.



## 1.4 Διορθώθηκε το παλιό bug στο k-step score mapping

Το παλιό πρόβλημα ήταν ότι γινόταν:

```python
score = alpha * sims[i] + (1 - alpha) * graph_score
```

όπου το `i` ήταν index στους neighbors, όχι στο corpus.

Στη νέα έκδοση αυτό έχει βελτιωθεί. Πλέον υπάρχει `sim_scores_for_query(query_id, ids)` που επιστρέφει dictionary:

```python
sim_dict = dict(zip(documents, sims))
```

και στο graph-aware reranking χρησιμοποιείται:

```python
score = alpha * sim_dict[node_id] + (1 - alpha) * graph_score
```

Αυτό διορθώνει τη βασική λογική αντιστοίχισης `node_id → cosine similarity`.



## 1.5 Το shortest path refactor κινείται επίσης σωστά

Το `shortest_path` πλέον δεν κάνει απευθείας `sims[i]` πάνω στα path nodes. Χρησιμοποιεί και αυτό reranker logic (`graph_aware`, `cross_encoder`, `BM25`). Άρα το αρχικό indexing bug έχει σε μεγάλο βαθμό διορθωθεί και εκεί.



## 1.6 Έχει προστεθεί πειραματική υποδομή για rerankers

Προστέθηκαν:

```text
graph_aware reranker
BM25 hybrid reranker
cross_encoder reranker
```

Αυτό δείχνει ότι ο πως έχει γίνει προσπάθεια να επεκταθεί η σύγκριση πέρα από απλό PPR/k-step. Είναι θετικό ως υποδομή, αλλά πρέπει να μπει προτεραιότητα: πρώτα σταθεροποιούμε τα graph-based methods και μετά κρατάμε τα υπόλοιπα ως extra.



## 1.7 Έχει βελτιωθεί το graph analysis

Στο `plotting.py` πλέον υπάρχουν επιπλέον graph statistics όπως:


Αυτό είναι πολύ θετικό, γιατί βοηθά να συνδεθεί η δομή του γράφου με την απόδοση στο retrieval.



## 1.8 Διορθώθηκε η ορολογία στα plots

# 2. Σημαντικά προβλήματα που παραμένουν

## 2.1 Πολύ σοβαρό bug: έχει μπερδευτεί το nDCG με το MAP στα saved results

Στα experiment functions γίνεται σωστά η συλλογή:

```python
ndcg_scores.append((query_id, ev.nDCGk_score(predictions, correct, k)))
avg_precisions.append((query_id, ev.avg_precision(predictions, correct, k)))
```

Αλλά μετά αποθηκεύεται ανάποδα:

```python
eval_scores["ndcg"] = avg_precisions
eval_scores["avg_precisions"] = ndcg_scores
```

Αυτό σημαίνει ότι στο `evaluate_method`:

```python
ndcg_total = scores["ndcg"]
avg_prec_sum = scores["avg_precisions"]
```

το `ndcg` γίνεται στην πράξη MAP και το `mapk` γίνεται στην πράξη nDCG.

Αυτό είναι κρίσιμο. Πρέπει να διορθωθεί άμεσα σε όλες τις search functions.

### Σωστό:

```python
eval_scores["ndcg"] = ndcg_scores
eval_scores["avg_precisions"] = avg_precisions
```

Αν δεν διορθωθεί, οι τελικοί πίνακες θα έχουν λάθος στήλες.



## 2.2 Το PPR δεν κάνει ακόμη πλήρες graph-wide retrieval

Η `personalised_pagerank` τρέχει PageRank πάνω στον γράφο, αλλά στο τέλος κάνει ranking μόνο πάνω στα αρχικά `ids`:

```python
for i, node_id in enumerate(ids):
    score = alpha * sim_dict[node_id] + (1 - alpha) * graph_scores[node_id]
    final_scores.append((node_id, score))
```

Αυτό σημαίνει ότι το PPR λειτουργεί περισσότερο σαν reranking των αρχικών seed documents και όχι σαν πλήρης graph propagation που μπορεί να φέρει νέα passages από τον γράφο.

Αν `init = 5` και `k = 10`, τότε η PPR μέθοδος δεν μπορεί να επιστρέψει πραγματικά 10 διαφορετικά documents, γιατί κοιτάει μόνο τα 5 αρχικά seed ids.

### Τι πρέπει να αποφασιστεί

Υπάρχουν δύο επιλογές:

#### Επιλογή Α — PPR ως graph-wide retrieval

Να γίνει ranking σε όλα τα graph nodes:

```python
for node_id in graph.nodes():
    score = alpha * sim_dict.get(node_id, 0.0) + (1 - alpha) * graph_scores[node_id]
```

#### Επιλογή Β — PPR ως reranker

Αν θέλουμε PPR μόνο ως reranker, τότε πρέπει να δηλώνεται καθαρά:

```text
PPR reranking over top-init cosine results
```

και να ισχύει:

```text
init >= k
```

Αλλιώς δεν είναι δίκαιη σύγκριση με cosine baseline@K.



## 2.3 Το PPR ακόμη δεν έχει safety checks

Στο `ppr_graph_scores` και στην `personalised_pagerank` υπάρχει ακόμη:

```python
total = sum(similarities)
weights = {
    node_to_idx[node_id]: similarities[i] / total
    for i, node_id in enumerate(ids)
    if node_id in node_to_idx
}
```

Πρέπει να ελεγχθούν:

```text
τι γίνεται αν total == 0
τι γίνεται αν κανένα seed node δεν υπάρχει στον graph
τι γίνεται αν το weights dictionary βγει κενό
```

### Προτεινόμενη διόρθωση

```python
valid = [
    (node_id, similarities[i])
    for i, node_id in enumerate(ids)
    if node_id in node_to_idx
]

total = sum(score for _, score in valid)

if total == 0 or len(valid) == 0:
    return {}
```



## 2.4 Το threshold grid εξακολουθεί να έχει typo

Στο `main.py` υπάρχει ακόμη:

```python
threshold_grid = {
    "threshold_distance ": [...]
}
```

με κενό στο τέλος του key.

Όμως η graph construction function περιμένει:

```python
threshold_params["threshold_distance"]
```

χωρίς κενό.

### Πρέπει να γίνει:

```python
threshold_grid = {

```



## 2.5 Το threshold graph δεν φαίνεται να έχει μπει πλήρως στο `make_graphs`

Υπάρχει manual επιλογή για threshold graph στο interactive mode, αλλά στο `make_graphs()` δεν φαίνεται να υπάρχει καθαρό loop για threshold graph όπως υπάρχει για kNN, mutual kNN και clustering.

Πρέπει να προστεθεί:

```python
for threshold_combo in grid_combinations(threshold_grid):
    threshold_params = merge(BASE_THRESHOLD, threshold_combo)
    name = make_name(f"threshold_{pre_tag}", threshold_params, graph_params)
    ex.threshold_graph(c, threshold_params, preprocess, name, graph_params, True)
```



## 2.6 Υπάρχει πιθανό signature mismatch στο threshold wrapper

Η `build_threshold_graph` πλέον παίρνει:

```python
build_threshold_graph(corpus, threshold_params, preprocess, name, graph_params, save)
```

αλλά το wrapper `threshold_graph` στο `experiments.py` φαίνεται να καλεί:

```python
gc.build_threshold_graph(sampled_items, threshold_params, preprocess, name, save)
```

δηλαδή χωρίς `graph_params`.

Αν κληθεί έτσι, θα σπάσει.

Πρέπει να αλλάξει σε:

```python
def threshold_graph(sampled_items, threshold_params, preprocess, name, graph_params, save):
    gc.build_threshold_graph(sampled_items, threshold_params, preprocess, name, graph_params, save)
```



## 2.7 Το `make_name` έχει αλλάξει signature, αλλά το `make_graphs` φαίνεται να το καλεί με παλιά μορφή

Η `make_name` πλέον ορίζεται ως:

```python
def make_name(prefix, varied: dict, graph: dict) -> str:
```

αλλά μέσα στο `make_graphs()` φαίνεται να καλείται έτσι:

```python
name = make_name(f"knn_{pre_tag}", knn_combo)
```

δηλαδή με δύο arguments αντί για τρία.

Αυτό πιθανότατα θα προκαλέσει error.

Πρέπει είτε:

```python
name = make_name(f"knn_{pre_tag}", knn_combo, graph_params)
```

είτε να δοθεί default value στη `make_name`.



## 2.8 Τα graph_params από input είναι strings, όχι booleans

Στο interactive input:

```python
DIRECTED = str(input("Directed graph? (True/False): "))
WEIGHTED = str(input("Weighted graph? (True/False): "))
graph_params = {"Directed": DIRECTED, "Weighted": WEIGHTED}
```

Αυτό αποθηκεύει strings `"True"` / `"False"`, όχι boolean `True` / `False`.

Όμως στο `graph_construction.py` γίνεται έλεγχος τύπου:

```python
if graph_params["Directed"] == False:
```

Αν το value είναι `"False"` string, τότε:

```python
"False" == False
```

είναι `False`.

Άρα μπορεί να φτιάχνεται directed graph ακόμη και όταν ο χρήστης έβαλε False.

### Πρέπει να γίνει:

```python
DIRECTED = input("Directed graph? (True/False): ").strip() == "True"
WEIGHTED = input("Weighted graph? (True/False): ").strip() == "True"
```

ή ακόμη καλύτερα με helper parser.



## 2.9 Το BM25 reranker έχει πιθανό bug με το text indexing

Στο `rerank_hybrid` υπάρχει:

```python
texts = [c_text[id][0] for id in ids]
```

Αν το `c_text[id]` είναι string passage, τότε `c_text[id][0]` είναι μόνο ο πρώτος χαρακτήρας του passage, όχι ολόκληρο το passage.

Άρα το BM25 μπορεί να δουλεύει πάνω σε μεμονωμένους χαρακτήρες αντί για κανονικά κείμενα.

### Πρέπει να γίνει:

```python
texts = [c_text[id] for id in ids]
```

Επίσης το:

```python
embs = [c[id][0] for id in ids]
```

δεν φαίνεται να χρησιμοποιείται και μπορεί να αφαιρεθεί.



## 2.10 Το CrossEncoder φορτώνεται μέσα σε κάθε query

Στο `rerank_cross_encoder` φορτώνεται:

```python
cross_encoder = CrossEncoder('cross-encoder/ms-marco-MiniLM-L-6-v2')
```

μέσα στη function.

Αυτό σημαίνει ότι αν τρέξουν πολλά queries, το μοντέλο θα φορτώνεται ξανά και ξανά, κάτι που είναι πολύ αργό.

Πρέπει να φορτώνεται μία φορά και να περνάει ως parameter ή να γίνεται global/cache.
